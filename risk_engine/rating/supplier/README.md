# 代理商综合评价体系

> 版本: v3.2 | 最后更新: 2026-05-28
> 所属模块: `risk_engine/rating/supplier/`

---

## 一、业务背景

湖南省保障金业务模式中，需要根据代理商的综合质量来评定等级（A/B/C），等级决定了：

| 评级 | 保障金缴纳比例 | 每月可办单数 |
|:----:|:-------------:|:-----------:|
| A | 60% | 50 笔 |
| B | 70% | 40 笔 |
| C | 80% | 30 笔 |

评级越高的代理商，缴纳的保障金比例越低、可办单数越多。这是一个**用评级换额度、用额度控风险**的机制。

---

## 二、评级依据

评级综合了三个来源的数据：

### 2.1 翼支付门店综合评级
翼支付官方对门店的A/B/C/D/E评级，作为外部参考输入。无评级时用逾期质量估算。

### 2.2 淘顺实时授信的质态和规模
从`dws.dws_credit_yzf_order_complete`宽表及各业务表中提取：
- 逾期质量（订单逾期率）
- 客群结构（老客占比、融合占比、本网占比）
- 规模体量（交易笔数、门店数）
- 展业稳定性（活跃月数、最近活跃时间）
- 通过率异常（风控通过率的偏离度）

### 2.3 企查查企业工商信息（v3.0 新增）
从`risk_control.supplier_qichacha`表读取：
- **企业正规度**：登记状态（存续/注销）、企业类型（有限公司/个体户）
- **资本实力**：注册资本

## 三、评级方法

### 3.1 评分流程

```
提取原始维度数据 (extract.py)
        ↓
合并企查查数据 (run.py _merge_qichacha)
        ↓
各维度评分映射 0-100 分 (score.py)
        ↓
动态权重加权汇总为综合评分 (score.py)
        ↓
按排名百分比分 A/B/C (rate.py)
        ↓
写入本地库 supplier_evaluation (run.py)
```

### 3.2 维度权重

权重根据数据覆盖情况**动态调整**（以下为全量数据时的基准）：

| 维度 | 权重 | 数据来源 | 说明 |
|:----|:---:|:---------|:-----|
| 翼支付评级 | **28%** | 本地库/翼支付 | **v3.1升级：从11%提升至28%**，外部强信号 |
| 逾期质量 | **32%** | StarRocks | 最核心指标，直接反映资产质量 |
| 客群结构 | 10% | StarRocks | 老客/融合/本网占比综合 |
| 规模体量 | 9% | StarRocks | 交易笔数百分位排名 |
| 展业稳定性 | 7% | StarRocks | 活跃月数 + 最近活跃度 |
| 企业正规度 | **5%** | **企查查** | 登记状态+企业类型 |
| 资本实力 | **4%** | **企查查** | 注册资本 |
| 通过率异常 | 5% | StarRocks | 风控通过率偏离省均值程度 |

#### 权重动态调整规则

| 数据情况 | 调整方式 |
|:---------|:---------|
| **有翼支付评级** | 翼支付评级占 **28%**（强信号重点评估） |
| **无翼支付评级** | 翼支付评级28%权重按比例分给其他维度，**不影响总分** |
| **有企查查数据** | 企业正规度5%+资本实力4%正常参与评分 |
| **无企查查数据** | 9%权重按比例分给其他维度，**不影响总分** |

### 3.3 企查查维度评分规则

#### 企业正规度（权重6%）

| 条件 | 得分调整 |
|:----|:--------:|
| 基础分 | 60分 |
| 登记状态=存续 | +20分 |
| 登记状态=注销/吊销 | **-30分** |
| 企业类型=有限责任公司 | +15分 |
| 企业类型=股份有限公司 | +10分 |
| 企业类型=个人独资企业 | -5分 |
| 个体工商户 | 不调整 |
| 无企查查数据 | **50分（中性，权重自动重分配）** |

#### 资本实力（权重5%）

注册资本对数映射：20 + 80 × log₁₀(资本/1000)

| 注册资本 | 得分 |
|:--------|:----:|
| 0万 | 20分 |
| 10万 | 40分 |
| 50万 | 60分 |
| 200万 | 80分 |
| 1000万+ | 100分 |
| 无企查查数据 | **50分（中性，权重自动重分配）** |

### 3.4 动态权重重分配（核心设计）

权重根据两个条件动态调整：**是否有翼支付评级** 和 **是否有企查查数据**。

```
有翼支付+有企查查 → 8个维度全量参与（翼支付28%主导）
有翼支付+无企查查 → 企查查2个维度剔除，其余6个维度重分配
无翼支付+有企查查 → 翼支付剔除，其余7个维度重分配（逾期质量占比提升）
无翼支付+无企查查 → 翼支付+企查查剔除，逾期质量成为核心
```

**效果验证：**
| 场景 | 平均分 | 差异 |
|:----|:-----:|:----:|
| 有翼支付评级 | **71分** | — |
| 无翼支付评级 | **49分** | 差距合理（翼支付本身是强信号） |
| 有企查查 | 73分 | — |
| 无企查查 | 72分 | **仅1分 ✅** |

> **核心原则：有数据就重点评，没数据不影响总分。**

### 3.5 各维度评分映射规则

| 指标 | 方向 | 映射方式 | 示例 |
|:----|:---:|:--------|:-----|
| 逾期率 | 越小越好 | 反向线性：0%→100分, 10%→0分 | 2%→80分 |
| 老客占比 | 越高越好 | Floor线性：0%→30分, 100%→100分 | 60%→72分 |
| 融合占比 | 越高越好 | Floor线性：0%→30分, 100%→100分 | 80%→86分 |
| 交易笔数 | 越多越好 | 百分位排名 | top 1%→100分 |
| 翼支付评级 | 等级映射 | A→100, B→80, C→60, D→40, E→20 | |
| 活跃月数 | 越多越好 | 线性：6月→100分 | |
| 通过率偏离 | ±0 | >0→100分, <0→按差距扣分 | |

### 3.6 无翼支付评级处理

没有翼支付评级的代理商（外省全部+湖南部分），28%的翼支付权重**自动按比例分配到其他维度**，其中逾期质量受益最大（32%→约45%），确保**实际资损表现成为核心评判依据**。

### 3.7 数据不足处理

对于 **交易<20笔 或 活跃<2月 或无到期订单** 的代理商，标记为"数据不足"：

- 逾期率为0时 → 不给100分，**默认50分**（原因：还没到还款期，无法确认真实表现）
- **从严不放松**：对数据不足的代理商标记风险，不因数据缺失而虚高评分

### 3.8 空值处理

| 缺失维度 | 处理方式 |
|:--------|:---------|
| 翼支付评级 | 权重自动分配给其他维度（主要是逾期质量），不影响总分 |
| 客群子维度 | 缺少数值时补 0.5（中性值），最低保底30分 |
| 企查查数据 | 企业正规度+资本实力权重自动分配给其他维度，不影响总分 |
| 门店质量 | 当前未参与评分，字段保留 |

### 3.9 评级分配

```
综合评分按排名百分比分配：
  A级: 前 1%      (评分最高的一批)
  B级: 1% ~ 20%   (评分较好的)
  C级: 剩余 80%   (其余)
```

### 3.10 办单量硬门槛（v3.2 新增）

百分位排名后，进一步按**近12个月办单数**执行硬门槛降级：

| 评级 | 最低办单数 | 不满足时 |
|:----:|:---------:|:--------|
| A 级 | 50 单 | 降为 B 级 |
| B 级 | 20 单 | 降为 C 级 |

**效果：** 防止评分高但实际业务量极少的代理商拿到高评级（如某省只有10家代理商，前1%÷按百分位只有1家A，但这个A可能才办了几单）。

阈值可在 `config.py` 的 `A_MIN_ORDERS` / `B_MIN_ORDERS` 调整。

**降级链：** A级不满50单→B级，再检查B级是否满20单→不满则继续降C，确保结果经过两层过滤。

## 四、数据模型

### 4.1 核心表: `supplier_evaluation`

| 字段 | 类型 | 说明 |
|:----|:----|:-----|
| `supplier_id` | VARCHAR(32) PK | 代理商唯一编码 |
| `supplier_name` | VARCHAR(128) | 代理商名称（v2.0 新增） |
| `province` | VARCHAR(64) | 省份 |
| `compliance_score` | DECIMAL(5,2) | 综合评分 0-100 |
| `supplier_rating` | VARCHAR(4) | 最终评级 A/B/C |
| `num_overdue_rate` | DECIMAL(6,5) | 订单逾期率 |
| `risk_pass_rate` | DECIMAL(6,5) | 风控通过率（剔除保证金/特批） |
| `total_transaction_count` | INT | 累计交易笔数 |
| `active_months` | INT | 活跃月份数 |
| `yzf_rating` | VARCHAR(16) | 翼支付评级 |
| 及其他 40+ 维度字段 | | 详见 `schema.py` / ORM 定义 |

### 4.2 辅助表: `supplier_qichacha`（v3.0 新增）

| 字段 | 类型 | 说明 |
|:----|:----|:-----|
| `supplier_name` | VARCHAR(256) PK | 代理商名称 |
| `register_status` | VARCHAR(32) | 登记状态（存续/注销/吊销） |
| `enterprise_type` | VARCHAR(64) | 企业类型（有限公司/个体户等） |
| `registered_capital` | VARCHAR(32) | 注册资本 |
| `paid_capital` | VARCHAR(32) | 实缴资本 |
| `credit_code` | VARCHAR(32) | 统一社会信用代码 |
| `enterprise_scale` | VARCHAR(16) | 企业规模（微型/小型/中型） |
| `insured_count` | VARCHAR(16) | 参保人数 |

### 4.3 辅助表: `yzf_supplier_rating`

| 字段 | 类型 | 说明 |
|:----|:----|:-----|
| `supplier_id` | VARCHAR(64) PK | 代理商ID |
| `level` | VARCHAR(4) | 翼支付评级 A/B/C/D/E |

### 4.4 架构设计（两处定义模式）

```
MySQL table: risk_control.supplier_evaluation  ← 唯一权威来源
         ↑                       ↑
         |                       |
risk_engine/rating/supplier/   backend/database/orm/
    schema.py  ← 写入侧契约    supplier_rating.py  ← 读取侧ORM
    (写入方定义)                 (读取方定义)

物理上分离，逻辑上对齐，不互相import。
```

## 六、运行方式

```python
# 全量跑（默认全国）
from risk_engine.rating.supplier.run import run_supplier_rating
df = run_supplier_rating(data_date="2026-05-25")

# 仅某省
df = run_supplier_rating(data_date="2026-05-25", province="湖南省")

# 只算不写库（用于调试）
df = run_supplier_rating(data_date="2026-05-25", write_to_db=False)
```

### 依赖数据

运行前确保本地 MySQL 以下表有数据：

| 表 | 数据来源 | 更新频率 |
|:---|:--------|:--------|
| `supplier_qichacha` | 企查查批量导出 | 按需 |
| `yzf_supplier_rating` | 翼支付评价表导入 | 按需 |
| `risk_parameter_pass_radio` | 人工配置 | 按需 |

### 定时更新

建议频率：**每月初**跑一次，更新 `data_date` 为最新月份。

## 七、结果验证

### 7.1 评级 × 核心指标趋势

```
评级    平均分    逾期率      交易笔数    活跃月
A级      96      0.05%       249         6.9
B级      88      0.30%       115         6.2
C级      51      1.94%        11         2.6
```

三条趋势线均正确：A > B > C，评级体系有效。

### 7.2 翼支付评级 × 我方评分

```
翼支付A级 → 平均83分
翼支付B级 → 平均80分
翼支付C级 → 平均76分
翼支付D级 → 平均69分
翼支付E级 → 平均62分
```

与外部评级高度一致，验证评分体系合理。

### 7.3 逾期率分段 × 评分

```
逾期 0%     → 平均 66 分 (含数据不足者被压低)
逾期 0-2%   → 平均 76 分
逾期 2-5%   → 平均 64 分
逾期 5-10%  → 平均 43 分
逾期 >10%   → 平均 23 分
```

逾期率越高分越低，区分度明确。

### 7.4 翼支付评级 × 我方评级（v3.1 权重28%）

| 翼支付评级 | 我方A级占比 | 我方B级占比 |
|:---------:|:----------:|:----------:|
| A级 | **47.3%** ⭐ | 51.5% |
| B级 | 0% | **85.9%** ✅ |
| C级 | 0% | 6.6% |
| D级 | 0% | 0% |
| E级 | 0% | 0% |

翼支付评级与我方评级高度一致，A级代理商的98.8%成为我方A/B级。

## 八、已知限制

| 限制 | 影响 | 改进方向 |
|:----|:----|:---------|
| 门店质量数据覆盖率~8.7% | 当前未参与评分 | 待门店管控完善后补充 |
| 翼支付评级仅覆盖湖南 | 外省翼支付权重自动重分配给逾期质量 | 争取获取更全评级数据 |
| 企查查数据仅覆盖湖南 | 外省企查查维度权重自动重分配 | 同步其他省企查查数据 |
| 单卡/融合仅湖南可区分 | 其他省份客群结构维度减项 | 按省份差异化处理 |
| 金额逾期率未计算 | 逾期质量只用了订单维度 | 后续加入 `overdue_amount` |
